AI 에이전트와 함께 코딩을 시작했다. 

처음에는 단순히 질문만 던지다가도, 
작업이 복잡해질수록 "어떻게 시켜야 더 똑똑하게 일할까?" 고민하게 된다. 

오늘은 윈도우 환경에서 Gemini CLI 같은 도구를 사용할 때, 
작업의 난이도에 따라 선택할 수 있는 **3가지 개발 워크플로우**를 정리해 보았다.

일단, 시작하기에 앞서 3가지 방법 중 어느것이 더 낫다 라는건 없다.
그때 그때 가장 적절한 방법을 쓰면 된다.

---

## 1. 방법 1: 다이렉트 프롬프트 (가장 빠른 방식)
**[프롬프트 -> 바로 개발]**

별다른 준비물 없이 터미널에서 바로 명령을 내리는 방식이다. 

- **장점:** 매우 빠르다. 사소한 버그 수정이나 간단한 리팩토링에 최적이다.
- **특징:** 
    - `GEMINI.md` 파일이 프로젝트 루트에 있다면, 
      에이전트가 이를 먼저 읽고 프로젝트 규칙을 파악한다.
    - 만약 규칙 파일이 없다면 에이전트가 모든 파일을 스캔해야 하므로 
      **토큰 비용**이 높아지고 속도가 느려질 수 있다.
- **추천 상황:** "이 함수의 에러 메시지만 한글로 바꿔줘" 같은 단발성 작업.

**[예시 프롬프트]**
> "assets/js/main.js 파일에 있는 날짜 출력 형식을 'YYYY년 MM월 DD일'로 변경해줘."

## 2. 방법 2: PRD 기반 개발 (기획 중심)
**[프롬프트 -> prd.md 생성 -> 개발]**

작업을 시작하기 전, **PRD(Product Requirements Document)** 즉 
'제품 요구사항 명세서'를 먼저 만드는 단계다.

- **방식:** 에이전트에게 "이런 기능을 만들고 싶은데 기획안(PRD)부터 써줘"라고 시킨다.
- **장점:** "무엇을 해야 하는지" 기능 리스트가 명확해진다. 
  기획을 먼저 확인하고 수정할 수 있어 큰 틀에서의 실수를 줄인다.
- **단점:** 기술적인 구현 세부사항(어떤 클래스를 쓸지 등)은 여전히 AI의 판단에 의존하므로, 
  결과물이 예상과 다를 수 있다.
- **추천 상황:** 새로운 페이지 추가나 기존 기능의 대대적인 확장이 필요할 때.

**[예시 프롬프트]**
> "블로그에 '포스트 좋아요' 기능을 넣고 싶어. 필요한 데이터 구조와 UI 요구사항을 포함해서 prd.md를 먼저 작성해줘."

## 3. 방법 3: SDD 기반 개발 (최고 품질 지향)
**[프롬프트 -> spec.md -> plan.md -> task.md -> 개발]**

가장 정밀한 방식인 **SDD(Specification-Driven Development)**, 즉 '명세 기반 개발'이다. 
각 단계를 쪼개어 에이전트와 협업한다.

1. **spec.md:** 기획 중심 기능 명세                                                                                                                                     │
2. **plan.md:** 구현 계획 기능 명세                                                                                                                                     │
3. **task.md:** 작은 단위 기능 task 명세

- **장점:** 입력/출력과 흐름이 명확하여 **원하는 코드 품질**을 얻을 수 있다.
- **단점:** 명세서를 작성하고 확인하는 시간 자체가 부담스러울 수 있다.
- **특징:** 일부 단계를 **하네스(Harness) 개발 방법론**과 연계하면 자동화된 검증까지 가능해진다.
- **추천 상황:** 핵심 비즈니스 로직 구현이나 유지보수가 매우 중요한 대규모 프로젝트.

**[예시 프롬프트]**
> "사용자 로그인 시스템을 JWT 방식으로 구현하려고 해. 먼저 기술 명세(spec.md)와 구현 계획(plan.md), 작업 리스트(task.md)를 단계별로 작성해줘."

## 개인적인 생각

- **규칙은 간결함이 핵심이다:** 
  GEMINI.md가 너무 길면 토큰 효율이 떨어지고 지시를 누락할 수 있다. 
  핵심 규칙 위주로 짧게 유지하는 것이 좋겠다.

- **적재적소의 유연함:** 
  단순 수정은 방법 1로, 큰 기능은 방법 3로 대응해야 한다.

- 본인이 개발자가 아니라면 방법 2 방법으로 의사소통에 이용해볼 수 있을 것 같다.

- 상황에 맞는 워크플로우 선택이 생산성을 결정한다.

- **결국 프로세스가 답이다:** 
  AI는 도구일 뿐, 일을 시키는 방식이 코드 품질과 내 퇴근 시간을 결정한다.
