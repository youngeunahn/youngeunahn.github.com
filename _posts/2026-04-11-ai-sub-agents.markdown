Gemini CLI를 사용하다가 GEMINI.md 파일에 프로젝트의 모든 컨벤션 및 유의사항이 길게 작성되는것을 보고 비효율적이라고 생각이 들었다.  
또한 모든 작업을 하나의 에이전트에게 시키는 것이 비효율적일 때가 있는것이라고 생각들었다.    

각각의 파트에서 내용을 따로 알고있는 전문가 팀을 만들 수 있는 기능을 접하게 되었고   
마치 팀 프로젝트에서 '기획자', '개발자', 'QA'가 나누어져 있듯이, AI에게도 특정 역할(Role)을 부여하여 협업할 수 있다고 한다.  
  
하나의 협업팀 같은 **서브 에이전트(Sub-agent)**를 정의하고 활용하는 방법을 정리했다.  

---

## 1. 서브 에이전트(Sub-agent)란?

서브 에이전트는 특정 작업에 최적화된 독립적인 AI 전문가 이다. 
메인 에이전트가 전체적인 흐름을 관리하고 의사결정을 내린다면, 서브 에이전트는 부여받은 특정 임무에만 집중하여 정밀한 작업을 수행한다.

### 멀티 에이전트(Multi-agent)와는 무엇이 다른가요?
'멀티 에이전트'가 여러 AI가 협력하는 **전체적인 팀**을 의미하는 넓은 개념이라면, '서브 에이전트'는 그 팀을 운영하는 **계층적 방식**을 뜻한다.  
Gemini CLI는 '팀장(메인 에이전트)'이 '전문가 팀원(서브 에이전트)'에게 작업을 적절히 배분하는 **계층형 멀티 에이전트 아키텍처**를 채택하고 있다.  
서브에이전트 만들고 요청했더니 알아서 서브 에이전트가 호출되는걸 볼 수 있었다.

### 왜 서브 에이전트가 필요한가요?
- **컨텍스트 최적화:** 관련 없는 파일이나 히스토리는 무시하고, 작업에 필요한 정보만 집중해서 처리하여 정확도를 높인다.
- **전문성 강화:** 특정 도구(예: 테스트 러너, 보안 스캐너, 특정 언어의 린터) 사용법에 특화되도록 설정할 수 있다.
- **토큰 및 비용 절약:** 불필요한 대화 맥락을 배제하고 핵심 작업만 수행하므로 훨씬 경제적이다.

## 2. 기본으로 제공되는 내장 서브 에이전트

Gemini CLI를 설치하면 바로 사용할 수 있는 '기본 전문가'들이 있다. 기본 Tool처럼 제공 된다.

- **`@codebase_investigator` (코드베이스 조사관):** 가장 강력한 에이전트 중 하나이다. 프로젝트의 전체 구조를 파악하고, 특정 기능이 어디에 구현되어 있는지, 의존성 관계는 어떻게 되는지 심층적으로 분석한다.
- **`@cli_help` (CLI 도우미):** Gemini CLI 자체에 대한 전문가이다. 특정 명령어 사용법, 설정 파일(`settings.json`) 구성 방법, 새로운 기능을 사용하는 법을 가장 잘 알고 있다.
- **`@generalist` (제너럴리스트):** 범용적인 작업을 수행한다. 대량의 파일을 수정하거나, 복잡한 쉘 명령어를 실행해야 할 때 메인 에이전트가 이 친구에게 일을 맡기곤 한다.

## 3. 서브 에이전트 설정 및 정의

서브 에이전트는 프로젝트의 설정 파일이나 시스템 명령을 통해 정의할 수 있다. 
각 에이전트에게는 명확한 이름, 설명, 그리고 구체적인 행동 지침(Instructions)이 필요하다.

### 예시: 테스트 코드 생성 전문가 에이전트
JUnit5와 Mockito를 사용하여 단위 테스트만 전문적으로 작성하는 에이전트 설정 예시이다.

```yaml
subagents:
  - name: "test-expert"
    description: "JUnit5와 Mockito를 사용하여 완벽한 단위 테스트를 작성하는 전문가입니다."
    instructions: |
      - 모든 테스트는 Given/When/Then 패턴을 따릅니다.
      - 경계값 분석을 포함하여 최소 5개 이상의 테스트 케이스를 생성합니다.
      - 코드 커버리지가 80% 이상이 되도록 구성합니다.
```

## 3. 서브 에이전트 호출 및 사용법

Gemini CLI에서 서브 에이전트를 사용하는 방법은 크게 두 가지가 있다.

### A. 수동 호출 (Direct Call)
채팅창에서 `@` 기호와 에이전트 이름을 입력하여 특정 전문가에게 직접 일을 시킬 수 있다. 
마치 슬랙(Slack)에서 팀원을 태그하는 것과 비슷하다.

**사용 예시:**
- `@codebase_investigator 이 프로젝트의 전체적인 폴더 구조와 주요 모듈의 역할을 분석해줘.`
- `@cli_help 새로운 커스텀 서브 에이전트를 등록하는 설정 방법을 알려줘.`

### B. 자동 호출 (Automatic Delegation)
메인 에이전트가 사용자의 요청을 분석하여, "이 작업은 내가 직접 하는 것보다 전문가에게 맡기는 게 낫겠다"라고 판단하면 스스로 서브 에이전트를 호출한다. 
예를 들어, "이 프로젝트의 모든 파일에 라이선스 헤더를 추가해줘"라고 요청하면 메인 에이전트가 `generalist`에게 대량의 파일 수정 작업을 위임하는 식이다.

## 4. 나만의 커스텀 서브 에이전트 만들기

Gemini CLI의 가장 강력한 점은 누구나 자신만의 전문가를 정의할 수 있다는 것이다. 
마크다운(`.md`) 파일 하나만 작성하면 새로운 에이전트가 탄생한다.

### 정의 방법
보통 `.gemini/subagents/` 폴더 내에 마크다운 파일을 생성하여 정의한다. 
파일명(예: `security-guard.md`)이 곧 에이전트의 호출 이름(`@security-guard`)이 된다.

**에이전트 정의 파일 예시 (`security-guard.md`):**
```markdown
# Security Guard Agent

## 에이전트의 성격 (Persona)
너는 코드의 보안 취약점을 찾아내고 수정 제안을 하는 보안 전문가야. 
항상 OWASP Top 10 기준에 맞춰 코드를 검토해.

## Capabilities
- 코드 내 하드코딩된 API 키 탐색
- SQL Injection 취약점 점검
- 의존성 보안 이슈 확인

## Instructions
- 모든 취약점은 위험도(High, Medium, Low)와 함께 보고해.
- 수정 가능한 구체적인 코드 예시를 반드시 포함해.
```

이렇게 만든 에이전트는 바로 `@security-guard`로 소환하여 보안 점검을 맡길 수 있다.

## 5. 심화 활용: 전문가 간의 협업 워크플로우 (Reviewer -> Fixer)

진정한 멀티 에이전트의 힘은 여러 전문가가 순차적으로 협업할 때 나타난다. 
예를 들어, **"보안 검토는 꼼꼼한 분석가에게 맡기고, 실제 수정은 손이 빠른 개발자에게 맡기는"** 분업 시스템을 구축할 수 있다.

### 1단계: 검토 전담 에이전트 (`security-reviewer.md`)
이 에이전트는 읽기 권한만 가지며, 문제점을 찾아 리포트를 작성하는 역할만 수행한다. 수정 권한이 없으므로 안전하게 검토에만 집중한다.

```markdown
---
tools:
  - read_file
  - grep_search
---
# Security Reviewer
너는 코드의 보안 취약점을 검토하는 전문가야. 
취약점이 발견되면 **'수정 전문가(@code-fixer)를 위한 작업 리스트'**를 작성해줘.
너는 절대로 코드를 직접 수정해서는 안 돼.
```

### 2단계: 작업 전담 에이전트 (`code-fixer.md`)
이 에이전트는 수정 권한을 가지며, 리포트를 전달받아 실제로 코드를 고친다.

```markdown
---
tools:
  - read_file
  - replace
  - write_file
---
# Code Fixer
너는 보안 취약점을 실제로 수정하는 전문가야. 
`Security Reviewer`가 작성한 리포트를 바탕으로 코드를 안전하게 수정해줘.
```

### 3단계: 협업 실행하기 (메인 에이전트에게 지시)
사용자는 메인 에이전트(팀장)에게 다음과 같이 두 전문가의 협업 프로세스를 한 번에 지시할 수 있다.

> **사용자:** "@security-reviewer로 이 프로젝트의 인증 로직을 검토하고 리포트 써줘. 만약 고칠 게 나오면 그 리포트를 @code-fixer에게 넘겨서 수정까지 끝내줘."

이처럼 **'권한이 다른 전문가들'**을 배치하고 메인 에이전트가 그 순서를 조율하게 함으로써, 코드의 안전성과 작업의 정확도를 동시에 확보할 수 있다.

## 6. 실전 활용 시나리오

1. **문서화 전담 에이전트:** 코드를 분석하여 JSDoc, JavaDoc 또는 README 파일을 항상 최신 상태로 유지한다.
2. **리팩토링 전문가 에이전트:** 적용된 언어 사양(Java 21, Python 3.12 등)을 적용하여 기존 코드를 현대화하고 가독성을 개선한다.

## 마치며

AI 에이전트를 단순한 '채팅 상대'가 아니라 전문성을 가진 '팀원'으로 대하기 시작하면 개발 생산성이 비약적으로 향상된다. 
여러분의 워크플로우에 딱 맞는 서브 에이전트 팀을 꾸려보자!
